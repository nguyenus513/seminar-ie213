# Chương 1. Giới thiệu đề tài

## 1.1. Lý do chọn đề tài

MERN Stack là một hướng phát triển web phổ biến vì sử dụng JavaScript xuyên suốt từ frontend đến backend. Trong MERN, MongoDB thường được dùng làm cơ sở dữ liệu nhờ mô hình document linh hoạt, dễ mở rộng và phù hợp với dữ liệu bán cấu trúc.

Tuy nhiên, khi dữ liệu tăng lên, các chức năng tìm kiếm, lọc và sắp xếp sản phẩm có thể chậm nếu database không có index phù hợp. Nếu không có index, MongoDB có thể phải quét toàn bộ collection để tìm document thỏa điều kiện. Trường hợp này thường thể hiện trong execution plan bằng stage `COLLSCAN`. Khi có index phù hợp, MongoDB có thể dùng `IXSCAN` để giảm số document cần kiểm tra.

Vì vậy, đề tài xây dựng một ứng dụng MERN e-commerce mini có chức năng tìm kiếm sản phẩm thật, đồng thời tích hợp dashboard benchmark để đo và phân tích hiệu năng truy vấn MongoDB trước và sau khi áp dụng indexing.

# Chương 2. Cơ sở lý thuyết

## 2.1. MERN Stack

MERN Stack gồm MongoDB, Express.js, ReactJS và Node.js. ReactJS đảm nhiệm giao diện người dùng, Express.js và Node.js đảm nhiệm REST API, còn MongoDB lưu trữ dữ liệu dạng document.

Trong đề tài này, hệ thống tuân thủ nghiêm MERN: MongoDB là database chính, Express.js là backend REST API, React Vite là frontend SPA và Node.js là runtime. Không sử dụng Next.js, NestJS, GraphQL, Prisma, Redis, Kafka hoặc microservices để tránh lệch phạm vi.

## 2.2. MongoDB Index

Index trong MongoDB là cấu trúc dữ liệu hỗ trợ truy vấn hiệu quả hơn. Nếu không có index phù hợp, MongoDB có thể phải đọc nhiều document trong collection. Khi có index, MongoDB có thể tra cứu dựa trên index thay vì quét toàn bộ dữ liệu.

| Loại index | Mục đích trong đề tài |
|---|---|
| Single-field index | Tối ưu query theo một field như `category` |
| Compound index | Tối ưu query nhiều điều kiện và sort |
| Text index | Tối ưu tìm kiếm từ khóa sản phẩm |
| Partial index | Chỉ index document đang active |
| TTL index | Tự xóa benchmark log sau một thời gian |
| Unique index | Đảm bảo email user không trùng |

## 2.3. Execution Plan và executionStats

MongoDB cung cấp `explain()` để xem query planner và thống kê thực thi truy vấn. Với mode `executionStats`, hệ thống có thể thu thập các chỉ số như thời gian thực thi, số document được kiểm tra, số index key được kiểm tra và số document trả về.

| Chỉ số | Ý nghĩa |
|---|---|
| `executionTimeMillis` | Thời gian MongoDB xử lý truy vấn |
| `totalDocsExamined` | Số document MongoDB đã kiểm tra |
| `totalKeysExamined` | Số index key MongoDB đã kiểm tra |
| `nReturned` | Số document trả về |
| `stage` | Giai đoạn thực thi chính trong winning plan |
| `winningPlan` | Kế hoạch thực thi được MongoDB chọn |

## 2.4. Các stage quan trọng

| Stage | Ý nghĩa |
|---|---|
| `COLLSCAN` | MongoDB quét toàn bộ collection |
| `IXSCAN` | MongoDB quét qua index |
| `FETCH` | MongoDB lấy document thật sau khi scan index |
| `SORT` | MongoDB thực hiện sắp xếp dữ liệu |
| `TEXT_MATCH` | MongoDB xử lý text search |

## 2.5. Compound Index và ESR Rule

ESR là nguyên tắc thiết kế compound index theo thứ tự Equality, Sort, Range. Nếu query có điều kiện bằng, sort và range, field equality nên đặt trước, field sort đặt tiếp theo và field range đặt sau.

Ví dụ query:

```js
db.products.find({
  category: "phone",
  price: { $gte: 5000000, $lte: 30000000 }
}).sort({ rating: -1 })
```

Index theo ESR:

```js
db.products.createIndex({ category: 1, rating: -1, price: 1 })
```

## 2.6. Trade-off của indexing

Index giúp tăng tốc truy vấn đọc nhưng cũng có chi phí: tốn dung lượng lưu trữ, làm insert/update/delete nặng hơn và có thể không hiệu quả nếu tạo sai query pattern. Vì vậy, index cần được thiết kế dựa trên truy vấn thực tế và kiểm chứng bằng execution plan.

# Chương 4. Kết luận và khuyến nghị

## 4.1. Kết luận

Đề tài đã xây dựng được một hệ thống MERN e-commerce mini kết hợp dashboard benchmark MongoDB. Hệ thống không chỉ trình bày lý thuyết về index mà còn cho phép người dùng thao tác trực tiếp với search sản phẩm, tạo/xóa index, chạy benchmark, xem execution plan và so sánh hiệu năng trước/sau tối ưu.

Các kết luận chính:

- Indexing là kỹ thuật quan trọng để tối ưu truy vấn đọc trong MongoDB.
- `explain("executionStats")` giúp đánh giá hiệu năng dựa trên số liệu trực tiếp từ database.
- `COLLSCAN` thường cho thấy truy vấn phải quét collection, dễ gây chậm khi dữ liệu lớn.
- `IXSCAN` cho thấy MongoDB sử dụng index, thường giúp giảm số document cần kiểm tra.
- Compound index cần thiết kế theo query pattern, đặc biệt khi query vừa filter vừa sort.
- ESR Rule hữu ích khi thiết kế compound index có equality, sort và range.
- Không nên tạo index tùy tiện vì index làm tăng chi phí lưu trữ và chi phí ghi.

## 4.2. Bài học rút ra

- Khi benchmark baseline cần tắt `MONGOOSE_AUTO_INDEX` để tránh Mongoose tự tạo index.
- Không nên chỉ đo API response time vì thời gian API còn phụ thuộc network, serialization và frontend.

- Text search cần text index; nếu chưa có text index MongoDB sẽ báo lỗi, đây là hành vi đúng cần giải thích khi demo.

Single-field index phù hợp với truy vấn lọc đơn giản, trong khi compound index hiệu quả hơn với các truy vấn có nhiều điều kiện hoặc có sắp xếp. Các truy vấn liên quan đến lịch sử đơn hàng theo người dùng, trạng thái đơn hàng theo thời gian và danh sách sản phẩm có sắp xếp thể hiện mức cải thiện rõ rệt nhất.

Nguyên nhân có thể đến từ việc truy vấn theo `category` có độ chọn lọc chưa cao, tức là số lượng document khớp với category vẫn lớn. Trong trường hợp query trả về một tỷ lệ lớn của collection, chi phí dùng index kết hợp `FETCH` document có thể không thấp hơn đáng kể so với `COLLSCAN`. Điều này cho thấy index không phải lúc nào cũng làm truy vấn nhanh hơn, mà hiệu quả phụ thuộc vào selectivity của field và query pattern thực tế.

Từ đó có thể kết luận rằng việc tối ưu MongoDB không nên dừng ở câu hỏi “có index hay không”, mà cần đánh giá “index nào phù hợp với query pattern nào”. Việc thiết kế index cần dựa trên dữ liệu thực tế, query pattern và phân tích execution plan.

Demo flow đề xuất khi thuyết trình:

Các câu phản biện cần chuẩn bị:

**Hỏi: Vì sao có index mà query không nhanh hơn?**  
Trả lời: Vì index chỉ hiệu quả khi field có độ chọn lọc tốt. Nếu category có quá nhiều document trùng nhau, MongoDB vẫn phải đọc nhiều document sau khi quét index. Khi đó chi phí `IXSCAN + FETCH` có thể không tốt hơn `COLLSCAN` nhiều. Vì vậy index phải được thiết kế theo query pattern và selectivity, không nên tạo index bừa bãi.

**Hỏi: Dựa vào đâu biết query tối ưu hơn?**  
Trả lời: Dựa vào `explain("executionStats")` của MongoDB, cụ thể là `stage`, `executionTimeMillis`, `totalDocsExamined`, `totalKeysExamined` và `nReturned`. Khi stage chuyển từ `COLLSCAN` sang `IXSCAN` và số document examined giảm mạnh thì có thể kết luận index giúp MongoDB truy vấn hiệu quả hơn.

**Hỏi: Vì sao chọn compound index?**  
Trả lời: Vì nhiều query thực tế không chỉ lọc một field mà còn lọc theo nhiều điều kiện và sort. Ví dụ đơn hàng thường query theo `userId` và sort theo `createdAt`, hoặc sản phẩm query theo `category` rồi sort theo `price`. Compound index phù hợp hơn single index trong các trường hợp này.

## 3.15. Đánh giá mở rộng nhiều chiến lược Indexing

Trong phần đánh giá mở rộng, hệ thống không chỉ so sánh hai trạng thái trước và sau khi tạo index, mà tiến hành benchmark nhiều chiến lược indexing khác nhau trên cùng một nhóm truy vấn đại diện. Các chiến lược được đánh giá bao gồm no index, single-field index, compound index, compound index sai thứ tự, compound index theo ESR, partial index, text index và covered query.

Cách đánh giá này giúp xác định không chỉ việc index có cải thiện hiệu năng hay không, mà còn chỉ ra loại index nào phù hợp với từng query pattern cụ thể. Đối với các truy vấn có filter kết hợp sort, compound index thường hiệu quả hơn single-field index. Đối với truy vấn có equality, sort và range, index theo quy tắc ESR cho kết quả tốt hơn do thứ tự field trong index phù hợp với cách MongoDB xử lý truy vấn.

| T2 | Products category + sort price | Có | Có | Có | Có | Có | Không |

Nhóm metric này giúp đánh giá đầy đủ hơn vì đôi khi thời gian thực thi dao động do cache, nhưng số document examined và stage trong execution plan vẫn phản ánh rõ cách MongoDB xử lý truy vấn.

Ngoài ra, báo cáo cũng đánh giá trade-off của index thông qua chi phí ghi và dung lượng index. Index giúp cải thiện hiệu năng đọc, nhưng nếu tạo quá nhiều index sẽ làm tăng thời gian insert/update/delete và tăng dung lượng lưu trữ.

# Chương 4. Nghiên cứu mở rộng về tối ưu index và AI/ML-assisted tuning

## 4.1. Tối ưu truy vấn truyền thống trong MongoDB

MongoDB Query Planner tạo ra các candidate plans và chọn winning plan dựa trên query shape, index hiện có và cost tương đối. Các stage như `COLLSCAN`, `IXSCAN`, `FETCH`, `SORT` và `TEXT_MATCH` cho biết MongoDB đang xử lý truy vấn theo cách nào. Trong đề tài, `explain("executionStats")` được dùng để đo trực tiếp `executionTimeMillis`, `totalDocsExamined`, `totalKeysExamined` và `nReturned`.

## 4.2. Thiết kế index dựa trên query pattern

Các loại index truyền thống được đánh giá gồm single-field index, compound index, compound index sai thứ tự, ESR index, partial index, text index và covered query. Với compound index, thứ tự field rất quan trọng. MongoDB khuyến nghị ESR guideline: equality fields đặt trước, sort fields tiếp theo và range fields sau cùng trong nhiều trường hợp.

Ví dụ query:

```js
products.find({
  category: "phone",
  price: { $gte: 5000000, $lte: 30000000 }
}).sort({ rating: -1 })
```

Theo ESR:

- `category` là Equality.
- `rating` là Sort.
- `price` là Range.

Index phù hợp:

```js
{ category: 1, rating: -1, price: 1 }
```

Để chứng minh không phải compound index nào cũng tốt, hệ thống bổ sung chiến lược index sai thứ tự như `{ price: 1, category: 1 }` và so sánh với `{ category: 1, price: 1 }`.

## 4.3. Automatic Index Advisor

Automatic Index Tuning là hướng nghiên cứu trong đó hệ thống tự phân tích workload, sinh candidate indexes, ước lượng lợi ích từng index và chọn cấu hình index phù hợp. MongoDB Atlas Performance Advisor là một ví dụ thực tế, có khả năng theo dõi slow queries và gợi ý index theo query shape.

Trong phạm vi đề tài, hệ thống xây dựng một mini Index Advisor ở tầng ứng dụng MERN. Hệ thống thu thập query pattern, execution plan, danh sách index hiện tại và kết quả benchmark, sau đó đưa ra khuyến nghị index. Các khuyến nghị này không được áp dụng mù quáng mà cần chạy benchmark validation.

## 4.4. AI-assisted Index Tuning

Trong phạm vi đề tài, AI được vận dụng ở tầng hỗ trợ ra quyết định thay vì thay thế cơ chế tối ưu truy vấn của MongoDB. Hệ thống AI Index Advisor nhận đầu vào là query pattern, execution plan, danh sách index hiện có và kết quả benchmark. Sau đó hệ thống đề xuất các index tiềm năng, giải thích lý do và cảnh báo trade-off. Để tránh phụ thuộc hoàn toàn vào AI, mọi đề xuất đều được kiểm chứng lại bằng benchmark trước/sau và `explain("executionStats")`.

Kiến trúc đề xuất:

```text
User search / Benchmark query
        ↓
MongoDB explain("executionStats")
        ↓
Extract query pattern + metrics
        ↓
Rule-based analyzer
        ↓
AI/ML advisor explanation
        ↓
Suggested indexes
        ↓
Benchmark validation
        ↓
Accept / Reject recommendation
```

Rule-based advisor hoạt động theo các nguyên tắc:

- Nếu query có filter đơn giản thì gợi ý single-field index.
- Nếu query có nhiều filter hoặc filter + sort thì gợi ý compound index.
- Nếu query có equality + sort + range thì gợi ý ESR index.
- Nếu query thường xuyên có `isActive=true` thì gợi ý partial index.
- Nếu projection fields có thể nằm hoàn toàn trong index thì gợi ý covered query.
- Nếu query là keyword search thì gợi ý text index.
- Nếu field có selectivity thấp thì cảnh báo index có thể không cải thiện mạnh thời gian.

## 4.5. ML-based Index Benefit Predictor

Trong phạm vi đề tài, Random Forest được đề xuất làm mô hình ML chính cho bài toán dự đoán chiến lược index. Lý do là dữ liệu benchmark có dạng tabular, gồm các đặc trưng như số field filter, số field sort, stage hiện tại, `totalDocsExamined`, `totalKeysExamined`, `nReturned` và loại index. Random Forest phù hợp với dữ liệu tabular nhỏ đến trung bình, dễ huấn luyện, ít yêu cầu tiền xử lý phức tạp và dễ giải thích hơn so với deep learning.

Mô hình ML không thay thế MongoDB Query Optimizer. Mục tiêu của mô hình là hỗ trợ ranking các chiến lược index tiềm năng. Sau khi ML đề xuất strategy, hệ thống vẫn phải tạo index và chạy benchmark thực nghiệm bằng `explain("executionStats")` để xác nhận hiệu quả.

Các feature dự kiến:

| Feature | Ý nghĩa |
|---|---|
| `hasEqualityFilter` | Query có equality filter không |
| `hasRangeFilter` | Query có range filter không |
| `hasSort` | Query có sort không |
| `filterFieldCount` | Số field filter |
| `sortFieldCount` | Số field sort |
| `indexFieldCount` | Số field trong index |
| `isCompoundIndex` | Có phải compound index không |
| `isTextIndex` | Có phải text index không |
| `isPartialIndex` | Có phải partial index không |
| `docsExaminedPerReturned` | Mức lãng phí quét document |
| `avgExecutionTimeMillis` | Target regression |

## 4.6. Learned Indexes, Reinforcement Learning và Vector Search

Các nghiên cứu mới cho thấy tối ưu index đang chuyển từ cách làm thủ công sang các phương pháp tự động và thông minh hơn. Automatic Index Tuning phân tích workload để sinh candidate indexes, ước lượng lợi ích và lựa chọn cấu hình index phù hợp. Machine Learning có thể được dùng để dự đoán chi phí truy vấn hoặc ranking các index strategy. Reinforcement Learning xem quá trình chọn index như một chuỗi hành động tối ưu dưới ràng buộc storage/write cost. Gần đây, LLM cũng được nghiên cứu như một công cụ hỗ trợ index tuning nhờ khả năng phân tích query và giải thích khuyến nghị.

Learned Indexes xem index như một mô hình ML học phân phối dữ liệu để dự đoán vị trí record. Tuy nhiên, trong phạm vi project MERN, learned index không được triển khai trực tiếp vì MongoDB không cho thay thế index nội bộ bằng mô hình tự xây ở tầng ứng dụng. Vector Search hoặc Semantic Search cũng là hướng mở rộng hấp dẫn cho trang mua hàng, nhưng không thay thế mục tiêu chính là benchmark MongoDB indexing.

## 4.7. Mức độ vận dụng AI trong đề tài

| Mức | Tên | Mô tả | Trạng thái trong đề tài |
|---:|---|---|---|
| 0 | No AI | Chỉ benchmark thủ công | Baseline |
| 1 | Rule-based Advisor | Gợi ý index theo ESR/query pattern | Đề xuất triển khai chính |
| 2 | ML Predictor | Random Forest dự đoán strategy/time | Mở rộng từ CSV benchmark |
| 3 | LLM-assisted Explanation | AI giải thích và cảnh báo trade-off | Optional |
| 4 | Semantic Search | Vector search cho sản phẩm | Future work |
| 5 | Learned Index | ML thay thế index truyền thống | Chỉ nghiên cứu |

Kết luận của phần nghiên cứu là AI/ML có tiềm năng hỗ trợ index tuning, nhưng không nên thay thế hoàn toàn công cụ tối ưu truyền thống nếu không có quy trình validation chặt chẽ. Trong hệ thống này, kết luận cuối cùng vẫn dựa trên benchmark thực nghiệm bằng `explain("executionStats")`.

## Phụ lục C. Kết quả Matrix Benchmark mở rộng

Để đánh giá sâu hơn, hệ thống đã chạy thêm matrix benchmark cho query `PRODUCTS_CATEGORY_SORT_PRICE`. Mục tiêu là so sánh nhiều chiến lược index trên cùng một query thay vì chỉ so sánh before/after.

Dataset: `30,000 products`, `80,000 orders`, `10,000 users`.  
Query: `products.find({ category: "phone" }).sort({ price: 1 })`.  
Iterations: `3`.  
Limit: `500`.

| Strategy | Stage | Avg Time ms | Docs Examined | Keys Examined | Docs/Returned | Faster vs NO_INDEX | Docs Reduced |
|---|---|---:|---:|---:|---:|---:|---:|
| NO_INDEX | SORT | 31.33 | 30,000 | 0 | 60 | 1x | 0% |
| PRODUCT_CATEGORY_SINGLE | SORT | 16.67 | 12,000 | 12,000 | 24 | 1.88x | 60% |
| PRODUCT_PRICE_CATEGORY_WRONG_ORDER | FETCH | 2.33 | 1,298 | 1,298 | 2.6 | 13.45x | 95.67% |
| PRODUCT_CATEGORY_PRICE | FETCH | 1 | 500 | 500 | 1 | 31.33x | 98.33% |
| PRODUCT_COVERED_LISTING | FETCH | 0.67 | 500 | 500 | 1 | 46.76x | 98.33% |

Nhận xét:

- `NO_INDEX` phải xử lý `SORT` và quét 30,000 documents, là baseline chậm nhất trong lần đo này.
- `PRODUCT_CATEGORY_SINGLE` giảm số document examined còn 12,000 nhưng vẫn còn `SORT`, cho thấy single-field index chưa đủ tốt cho query vừa filter vừa sort.
- `PRODUCT_CATEGORY_PRICE` phù hợp hơn vì thứ tự index hỗ trợ cả filter theo `category` và sort theo `price`, giảm docs examined xuống 500.
- `PRODUCT_COVERED_LISTING` cho thời gian thấp nhất trong lần đo matrix, nhưng cần lưu ý covered query chỉ hiệu quả khi projection chỉ lấy các field đã nằm trong index.
- Kết quả này củng cố kết luận: tối ưu MongoDB không chỉ là có index hay không, mà là chọn đúng loại index cho query pattern.
